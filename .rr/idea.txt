Click-to-view tool output in the Mission Control dashboard

OUTCOME
Make the tool-event rows in an agent's message stream clickable. Clicking a
row like `Read (ok, 0.04s)` opens a modal showing the actual output that tool
produced. Today the dashboard only ships `args_summary` (redacted, 256B cap)
and `error_summary`; the real tool output is observed by the PostToolUse hook
and then discarded. Tool outputs are the highest-bandwidth observability signal
a coordinator has into what a teammate just did — "Read returned 0.04s" tells
you almost nothing; the file contents tell you everything. This closes a real
blind spot in the coordinator-in-the-loop observability model, which is the
product's moat.

SHAPE (implementation direction — the planner should validate against the code)
- Capture tool output at the `sdk_teammate.py` PostToolUse hook (where outputs
  are currently seen and discarded).
- Store outputs in a bounded, in-memory, per-teammate structure (~50 events
  max, evicting oldest). Do NOT add output bodies to the frozen `ToolEvent`
  dataclass and do NOT push them inline on every broker snapshot / WebSocket
  state push — outputs are too large for that. Use a parallel store keyed by
  (teammate_id, tool_use_id).
- Serve via a new lazy-fetch HTTP endpoint on `ui_server.py`:
  `GET /tool-output/<teammate_id>/<tool_use_id>`. Localhost-only bind, same
  security posture as the just-shipped `/wait-messages` endpoint. Returns the
  (redacted, truncated) output body for that one tool invocation on demand.
- Dashboard render: make the tool-event row (the MiniMessage rows produced by
  `_format_tool_event_body` in ui_server.py) clickable; clicking opens a
  modal in the ConfigDetailPanel style showing the output body, with a
  copy-to-clipboard control.

SECURITY — THE CRUX (decision already made by the operator)
This is the FIRST time tool OUTPUT content crosses the broker boundary. Outputs
are unbounded and can contain anything: shell stdout, file contents, web
responses — including secrets. The existing `redaction.py` is args-only and
assumes a known small input shape per tool; it is NOT sufficient for outputs.
- Redaction-failure policy: MASK matched secrets inline (e.g. `AKIA[REDACTED]`),
  showing the surrounding non-secret text. Patterns to cover at minimum: AWS
  access keys (`AKIA...`), OpenAI-style keys (`sk-...`), GitHub tokens
  (`ghp_...`), bearer tokens, PEM private keys (`-----BEGIN PRIVATE KEY-----`),
  AWS session tokens. FAIL LOUD (emit a clear marker line, never silently pass
  raw) if the redactor itself raises.
- Hard truncation cap on stored/served output (~4KB suggested).
- A redaction bug here is a credential leak. The redaction patterns must be
  tested hard (happy + sad), and a sentinel security pass is mandatory before
  merge.

VALIDATION EXPECTATION
Tests at two layers (capture/redaction logic AND the HTTP endpoint), happy and
sad paths at each, per the project's validate-before-change standard. Redaction
pattern coverage is the highest-priority test surface.

EXPLICITLY OUT OF SCOPE (do not plan these)
- Persistent on-disk JSONL of tool outputs (a separate, larger feature).
- Syntax highlighting of output bodies.
- Diff view between outputs.
- Any change to how `args_summary` / `error_summary` already work.

CONTEXT POINTERS FOR THE PLANNER
- `claude_crew/sdk_teammate.py` — PostToolUse hook (F8 tool tracking) is where
  output is currently seen and dropped.
- `claude_crew/teammate.py` — `ToolEvent` dataclass and the `_completed_tool_events`
  deque; understand why ToolEvent is frozen/small before deciding the store shape.
- `claude_crew/redaction.py` — existing args-only redaction (v1 allowlist:
  Bash, Task, WebFetch); the output redactor is new work alongside it.
- `claude_crew/ui_server.py` — `_format_tool_event_body`, the existing route
  registration in `_make_app`, and the localhost-bind posture in `serve()`; the
  `/wait-messages` endpoint added in the most recent feature is the reference
  pattern for a new localhost HTTP route.
- `claude_crew/broker.py` — `BrokerSnapshot` / `snapshot()` and how tool_events
  flow to the UI today.
- `claude_crew/ui/dashboard.html` — the dashboard front-end (MiniMessage rows,
  ConfigDetailPanel modal style to mirror).
- CLAUDE.md "validate the whole suite when changing widely-consumed behavior" —
  run the full `uv run pytest`, not a filtered subset, at the validation gate.
